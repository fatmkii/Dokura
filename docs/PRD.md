# 项目简介
 - Dokura，一个适用于管理Nas上大量同人志，并可以在安卓apk应用中快速浏览的项目。
 - 关键词：轻量化，高性能，无需外网
 - 首版Android、Web界面和项目文档只提供简体中文，不实现语言切换或自动翻译。文件名、tag和ZIP内路径保留原始Unicode文本，不翻译或转写；日期、文件大小和数字按简体中文格式显示。界面文案在代码中集中管理，避免散落硬编码。

# 项目架构
 ## 服务端
  - 长时运行的后台服务。实现文件监控处理以及管理用的web界面
  
 ## 客户端
  - 安卓apk应用。连接服务端，浏览同人志。

# 项目需求
 ## 基础需求边界
  - 同人志是大量.zip文件，内含大量png或jpg图片。在安卓的客户端端需要实现高性能的实时浏览。
  - 首版按10,000至100,000个ZIP的规模设计和验收，该范围不是硬数量上限；超过100,000后仍保持功能正确并继续发现文件，但不承诺达到首版性能目标。数量按当前Content中可见ZIP计算，不包含已删除但尚未清理的元数据。性能验收使用包含100,000个ZIP元数据、接近真实目录层级和tag分布的数据集；列表、搜索和分页可用少量真实ZIP配合代表性元数据及文件结构测试，真实ZIP处理性能另行测试。
  - 每个ZIP文件的预期大小约为10MB至1000MB，该范围是容量估算而非导入限制；小于10MB或大于1000MB的有效ZIP仍正常处理，首版不按压缩包文件大小设置硬上限。安全硬限制只采用ZIP内容处理章节规定的单条目解压大小、总解压大小和压缩比规则。性能测试至少覆盖接近1GB的ZIP，但不承诺任意超大文件具有相同响应速度。
  - 客户端与服务端之间采用HTTP/1.1 REST API通信，API以`/api/v1/`为前缀并通过OpenAPI描述。结构化数据使用JSON；图片和日志归档等文件使用独立HTTP接口返回二进制，不使用JSON或Base64封装。后台状态采用定时轮询，首版不使用WebSocket、SSE或gRPC。
  - 图片响应支持ETag和缓存控制；已落盘且可直接读取的缩略图及缓存文件支持HTTP Range并采用高效文件响应，ZIP内尚未缓存的原图边解压边流式返回。在千兆局域网且服务端存储、CPU及客户端不构成瓶颈的条件下，大文件连续传输目标达到100MB/s级别；首次从NAS读取并解压ZIP内图片不承诺达到该吞吐量。
  - 默认是单人使用，不需要多人协作功能。评分在服务端仅保存1份，供安卓客户端和Web端共享；浏览进度仅保存在安卓客户端本地，不跨设备同步，Web端不保存浏览进度。
  - 首版仅支持在线连接服务端浏览，客户端缓存只用于在线访问加速，不提供完整的离线书库功能。
  - 在100,000条元数据、单用户且服务端无其他高负载任务的验收环境中：已建立索引的列表、搜索、筛选和排序API的95%请求在500毫秒内返回；Android和Web在局域网中打开列表或详情且已有缩略图时，首屏在1秒内可操作；点击已生成的内容缩略图后300毫秒内显示缩略图占位，服务端在1秒内开始返回原图数据；Android缓存中的原图在翻页后300毫秒内显示；评分界面在300毫秒内乐观更新，正常局域网下1秒内获得服务端确认。
  - 上述性能指标不包含首次ZIP分析、首次缩略图生成、NAS休眠唤醒或网络中断恢复。验收报告记录CPU、内存、存储类型、网络条件和数据集。


# 技术栈共通约束
 - 首版开发基线为Python 3.14.x、FastAPI 0.138.x、Vue 3.5.x、Naive UI 2.44.x、Flutter 3.44.x stable、Node.js 24.x LTS、TypeScript 5.9.x和Vite 7.x；Dart使用Flutter 3.44.x附带的版本。
 - PRD记录兼容的版本系列，实际构建使用的精确版本由锁文件记录：Python依赖使用`uv.lock`，Web依赖使用包管理器锁文件，Flutter依赖使用`pubspec.lock`。Docker基础镜像固定到补丁版本和镜像摘要。开发期间可在同一minor系列内升级patch版本；minor或major升级必须单独完成兼容性验证。
